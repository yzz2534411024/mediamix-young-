## 1. 概述

### 1.1 文档目的

本文档针对视频播放器的核心功能链路，从**接口解析 → 视频解码 → 播放渲染 → 缓存加载**全流程进行系统性优化分析，提供可落地的技术方案与实施建议。

### 1.2 核心优化目标

| 指标 | 当前基线 | 优化目标 |
|------|---------|---------|
| 首屏加载时间 | 3~5s | ≤ 1.5s |
| 播放卡顿率 | 5%~8% | ≤ 1% |
| 卡顿恢复时间 | 2~4s | ≤ 0.5s |
| 解码帧率稳定性 | 波动 ±8fps | 波动 ±2fps |
| 缓存命中率 | 40%~60% | ≥ 85% |
| 内存峰值占用 | 300~500MB | ≤ 200MB |

### 1.3 优化原则

- **用户感知优先**：优化重点放在用户可感知的体验指标上
- **端云协同**：客户端与服务端配合，减少单点瓶颈
- **分级降级**：根据设备能力与网络状况自动降级，保证基本可用
- **数据驱动**：所有优化需有埋点数据支撑，避免盲目调优

## 2. 接口解析优化

### 2.1 问题分析

接口解析是视频播放链路的第一环节，主要瓶颈包括：

- **DNS 解析延迟**：首次请求 DNS 解析耗时 100~300ms
- **接口响应慢**：服务端视频信息接口 RTT 过长（500ms~2s）
- **重定向链路过长**：CDN 调度多次 302 跳转，增加首屏时间
- **解析失败重试策略不合理**：超时设置过长，重试无退避机制

### 2.2 优化方案

#### 2.2.1 DNS 预解析与缓存

```
策略：提前解析 + 本地缓存 + 过期刷新
```

| 措施 | 实现方式 | 预期收益 |
|------|---------|---------|
| DNS 预解析 | 在页面/App 启动时对视频域名执行 `dns-prefetch` | 减少 100~200ms |
| 本地 DNS 缓存 | 缓存域名→IP 映射，TTL 设为 300s | 避免重复解析 |
| HTTPDNS | 绕过运营商 LocalDNS，直连 HTTPDNS 服务 | 避免劫持与解析失败 |
| IP 直连 | 对已知 CDN 节点直接使用 IP 请求 | 省去 DNS 解析环节 |

**实现要点**：

```java
// Android 示例：DNS 预解析
public class DnsPrefetchManager {
    private static final long CACHE_TTL_MS = 300_000; // 5分钟

    public void prefetch(List<String> domains) {
        for (String domain : domains) {
            InetAddress.getAllByName(domain); // 触发系统DNS缓存
        }
    }

    public InetAddress resolve(String domain) {
        // 1. 先查本地缓存
        // 2. 缓存未命中走 HTTPDNS
        // 3. HTTPDNS 失败回退系统 DNS
    }
}
```

#### 2.2.2 接口预请求与并行化

```
策略：提前请求 + 并行化 + 数据预构建
```

- **预请求**：在用户进入视频详情页前（如列表页滑动时），提前请求视频信息接口
- **接口合并**：将视频信息、播放地址、广告配置等多接口合并为单接口，减少 RTT
- **并行请求**：无法合并的接口使用并发请求，总耗时取 max 而非 sum
- **GraphQL/BFF**：按需获取字段，减少冗余数据传输

**时序优化对比**：

```
优化前（串行）：
  列表点击 → 请求视频信息(500ms) → 请求播放地址(400ms) → 请求广告配置(300ms) → 开始播放
  总耗时：1200ms

优化后（并行+预请求）：
  列表滑动 → 预请求视频信息(已缓存) → 并行请求播放地址+广告配置(max(400,300)) → 开始播放
  总耗时：400ms
```

#### 2.2.3 CDN 调度优化

| 措施 | 说明 |
|------|------|
| 302 改为 307 | 保持 POST 请求体，减少重定向开销 |
| 客户端调度 | 服务端下发 CDN 列表，客户端根据测速结果选择最优节点 |
| 就近接入 | 基于 IP 地理位置选择最近 CDN 节点 |
| 备用线路 | 主 CDN 不可用时秒切备用 CDN |

**客户端 CDN 调度策略**：

```
1. 服务端返回多个 CDN 域名（主CDN + 备CDN）
2. 客户端对每个域名做 TCP Ping / HTTP 小文件测速
3. 选择延迟最低的 CDN 节点
4. 播放过程中持续监测下载速度，低于阈值时切换
5. 切换时从当前播放位置续播，使用新 CDN 地址重新请求
```

#### 2.2.4 超时与重试策略

```
超时配置：
  - 连接超时：3s（移动端）/ 2s（WiFi）
  - 读取超时：5s
  - 总请求超时：8s

重试策略（指数退避）：
  - 第1次重试：延迟 500ms
  - 第2次重试：延迟 1000ms
  - 第3次重试：延迟 2000ms
  - 最多重试 3 次，切换 CDN 后重置计数

降级策略：
  - 重试全部失败 → 尝试低清晰度播放地址
  - 低清晰度也失败 → 展示错误页，提供重试按钮
```

### 2.3 接口解析优化效果预估

| 优化项 | 节省时间 |
|--------|---------|
| DNS 预解析 + HTTPDNS | 100~200ms |
| 接口预请求 | 300~500ms |
| 接口并行化 | 200~400ms |
| CDN 调度优化 | 100~300ms |
| **合计** | **700~1400ms** |

## 3. 视频解码优化

### 3.1 问题分析

视频解码是 CPU/GPU 密集型操作，主要瓶颈：

- **软解码 CPU 占用高**：高分辨率视频软解码 CPU 占用可达 60%~80%
- **硬解码兼容性问题**：不同芯片平台硬解码能力差异大，部分格式不支持
- **解码初始化慢**：解码器创建与首帧解码耗时长
- **内存占用大**：解码缓冲区管理不当导致内存飙升

### 3.2 优化方案

#### 3.2.1 硬解码优先策略

```
策略：硬解优先 → 硬解失败自动降级软解 → 软解兜底
```

**硬解码能力探测表**：

| 编码格式 | 分辨率 | 硬解支持率 | 降级策略 |
|---------|--------|-----------|---------|
| H.264/AVC | 1080p | ≥ 98% | 直接硬解 |
| H.264/AVC | 4K | ≥ 85% | 检测后决定 |
| H.265/HEVC | 1080p | ≥ 70% | 检测后决定，不支持降级 H.264 |
| H.265/HEVC | 4K | ≥ 50% | 优先降级 1080p H.264 |
| AV1 | 1080p | ≥ 30% | 优先降级 H.265/H.264 |
| VP9 | 1080p | ≥ 60% | 不支持降级 H.264 |

**硬解码探测与降级实现**：

```java
public class DecoderStrategy {
    public DecoderType selectDecoder(VideoFormat format) {
        // 1. 查询设备硬解码能力
        CodecCapabilities caps = queryHardwareDecoder(format.codec, format.width, format.height);

        if (caps != null && caps.isSupported()) {
            // 2. 检查历史硬解失败记录
            if (!hasHistoricalFailure(format)) {
                return DecoderType.HARDWARE;
            }
        }

        // 3. 硬解不支持或历史失败，使用软解
        return DecoderType.SOFTWARE;
    }

    // 硬解失败时的降级处理
    public void onHardwareDecodeError(VideoFormat format, Exception e) {
        // 记录失败，避免反复尝试
        recordFailure(format);
        // 切换到软解码
        switchToSoftwareDecoder();
        // 如果是编码格式问题，请求降级清晰度
        if (isCodecIssue(e)) {
            requestFallbackStream();
        }
    }
}
```

#### 3.2.2 解码器复用池

```
策略：预创建解码器 + 对象池复用 + 延迟释放
```

- **预创建**：在视频信息获取后、数据下载前，提前创建解码器实例
- **对象池**：维护解码器对象池，切换视频时复用而非销毁重建
- **延迟释放**：解码器闲置后延迟 30s 释放，避免短时间切换视频时反复创建

```java
public class DecoderPool {
    private final Queue<MediaCodec> pool = new ConcurrentLinkedQueue<>();
    private static final int MAX_POOL_SIZE = 3;
    private static final long RELEASE_DELAY_MS = 30_000;

    public MediaCodec acquire(VideoFormat format) {
        MediaCodec decoder = pool.poll();
        if (decoder != null && isCompatible(decoder, format)) {
            decoder.flush(); // 重置状态
            return decoder;
        }
        if (decoder != null) {
            decoder.release(); // 不兼容则释放
        }
        return createDecoder(format);
    }

    public void release(MediaCodec decoder) {
        if (pool.size() < MAX_POOL_SIZE) {
            scheduleDelayedRelease(decoder); // 延迟释放
        } else {
            decoder.release();
        }
    }
}
```

#### 3.2.3 解码线程优化

| 优化项 | 实现方式 | 效果 |
|--------|---------|------|
| 解码线程绑大核 | 将解码线程绑定到 CPU 大核 | 提升 15%~25% 解码速度 |
| 优先级提升 | 设置解码线程为高优先级（THREAD_PRIORITY_URGENT_DISPLAY） | 减少调度延迟 |
| 避免线程竞争 | 解码线程独立，不与 IO/渲染线程共享锁 | 减少等待时间 |
| 异步解码 | 使用 MediaCodec 异步模式 | 避免同步等待，提升吞吐 |

#### 3.2.4 内存管理优化

```
策略：帧缓冲区动态管理 + 及时回收 + 内存压力感知
```

- **动态缓冲区**：根据分辨率与帧率动态计算缓冲区大小，避免固定分配过大内存
- **帧回收**：已渲染帧及时归还缓冲池，避免帧堆积
- **内存压力回调**：监听系统 `onTrimMemory`，在内存紧张时缩小缓冲区
- **低内存设备适配**：RAM < 2GB 设备限制最大缓冲帧数为 8 帧

**缓冲区大小计算**：

```
单帧大小 = width × height × 像素格式字节数（YUV420 = 1.5 bytes/pixel）
缓冲区大小 = 单帧大小 × 缓冲帧数

示例（1080p）：
  单帧 = 1920 × 1080 × 1.5 = 3,110,400 bytes ≈ 3MB
  缓冲 12 帧 = 36MB
  缓冲 24 帧 = 72MB

低内存设备：缓冲 8 帧 = 24MB
```

#### 3.2.5 低分辨率解码 + 超分渲染

在性能不足的场景下，使用低分辨率视频解码 + AI 超分辨率渲染：

```
适用场景：
  - 低端设备播放高分辨率视频
  - 设备发热降频时
  - 后台播放时

方案：
  1. 请求 480p/720p 码流进行解码
  2. 使用 GPU Shader / NPU 进行实时超分辨率
  3. 输出 1080p 画面

收益：
  - 解码性能降低 50%~70%
  - 画质损失可控（SSIM > 0.92）
  - 功耗降低 30%~40%
```

### 3.3 解码优化效果预估

| 优化项 | 性能提升 |
|--------|---------|
| 硬解码优先 + 自动降级 | CPU 占用降低 40%~60% |
| 解码器复用池 | 首帧解码时间减少 200~400ms |
| 解码线程优化 | 帧率稳定性提升 20% |
| 内存管理优化 | 内存峰值降低 30%~50% |
| 低分辨率+超分 | 低端设备可流畅播放高分辨率 |

## 4. 播放流畅性（无卡顿）优化

### 4.1 卡顿根因分析

```
卡顿分类与占比（典型场景）：

1. 缓冲不足卡顿（40%）：网速波动导致缓冲区耗尽
2. 解码慢卡顿（25%）：解码耗时超过帧间隔，丢帧
3. 渲染慢卡顿（15%）：UI 线程阻塞，VSync 丢帧
4. 内存抖动卡顿（10%）：GC 暂停导致帧延迟
5. 其他（10%）：线程竞争、锁等待等
```

### 4.2 优化方案

#### 4.2.1 自适应码率（ABR）

```
策略：基于带宽预测 + 缓冲区状态 + 设备能力的动态码率切换
```

**ABR 算法核心逻辑**：

```
输入：
  - 当前下载速度（滑动窗口平均）
  - 缓冲区时长（当前已缓冲可播放时长）
  - 设备解码能力（最大支持分辨率/帧率）
  - 用户偏好（手动选择清晰度时优先尊重）

输出：
  - 目标码率/分辨率

决策逻辑：
  if (缓冲时长 < 5s) {
    // 缓冲不足，降级
    targetBitrate = min(当前带宽 * 0.7, 当前码率 * 0.5)
  } else if (缓冲时长 > 30s && 带宽充裕) {
    // 缓冲充足且带宽够，升级
    targetBitrate = min(当前带宽 * 0.8, 设备最大支持码率)
  } else {
    // 维持当前码率
    targetBitrate = 当前码率
  }

切换规则：
  - 升级延迟：连续 5s 带宽充裕才升级（避免频繁切换）
  - 降级即时：带宽不足时立即降级（避免卡顿）
  - 切换间隔：两次切换至少间隔 10s
```

**带宽估计算法**：

```
采用滑动窗口 + EWMA（指数加权移动平均）：

  bandwidth = α × 最新采样带宽 + (1 - α) × 历史带宽

  α = 0.7（对最新数据更敏感，快速响应带宽变化）

  采样点：每下载一个 Segment 记录一次
  过滤：排除异常值（如 TCP 慢启动阶段的前 3 个采样）
```

#### 4.2.2 缓冲区管理优化

```
策略：动态缓冲 + 分级水位线 + 预加载
```

**缓冲区水位线设计**：

```
                    缓冲区状态图

  |----低水位----|----高水位----|----最大水位----|
  0s            5s            15s             30s

  低水位（5s）：
    - 触发：紧急下载，降级码率
    - 行为：以最大速度下载，降低清晰度

  高水位（15s）：
    - 触发：正常下载速度
    - 行为：按播放速度等速下载

  最大水位（30s）：
    - 触发：暂停下载，节省带宽
    - 行为：停止下载，直到缓冲降到高水位以下

  动态调整：
    - WiFi 环境：低水位 3s，高水位 20s，最大水位 60s
    - 4G 环境：低水位 5s，高水位 15s，最大水位 30s
    - 弱网环境：低水位 8s，高水位 10s，最大水位 15s
```

#### 4.2.3 Seek 优化

| 优化项 | 实现方式 | 效果 |
|--------|---------|------|
| 关键帧索引 | 预加载关键帧位置表，Seek 到最近关键帧 | Seek 时间减少 50% |
| 渐进式 Seek | 先跳到低分辨率关键帧快速显示，再加载高分辨率 | 用户感知 Seek 延迟降低 |
| 预测性预加载 | 根据用户行为预测 Seek 目标位置，提前缓冲 | 热门片段零延迟 |
| 后台解码 | Seek 后在后台解码到精确帧，前端显示最近关键帧 | 精确 Seek 无等待感 |

**关键帧索引预加载**：

```
服务端生成：
  - 每个视频生成关键帧索引文件（segment_time, keyframe_offset）
  - 索引文件大小通常 < 10KB

客户端使用：
  1. 视频信息获取时同时下载索引文件
  2. Seek 时查表定位最近关键帧
  3. 从关键帧开始解码到目标帧

  Seek(targetTime):
    keyframe = findNearestKeyframe(targetTime)  // O(logN) 二分查找
    seekToOffset(keyframe.offset)
    decodeFromKeyframe(targetTime - keyframe.time)
```

#### 4.2.4 渲染管线优化

```
策略：独立渲染线程 + SurfaceView + 帧节奏控制
```

- **SurfaceView 替代 TextureView**：SurfaceView 在独立图层渲染，不占用 UI 线程，减少 3~5ms 合成延迟
- **帧节奏控制**：使用 `Choreographer` 对齐 VSync，避免撕裂与丢帧
- **渲染线程隔离**：视频渲染在独立线程，不受 UI 操作影响
- **跳帧策略**：当解码帧堆积超过 5 帧时，跳到最新帧渲染，避免音视频不同步

```java
// 帧节奏控制示例
public class FramePacingController {
    private long lastFrameTimeNs = 0;
    private long frameIntervalNs; // = 1_000_000_000 / fps

    public boolean shouldRenderFrame(long timestampNs) {
        long elapsed = timestampNs - lastFrameTimeNs;
        if (elapsed >= frameIntervalNs) {
            lastFrameTimeNs = timestampNs;
            return true;
        }
        return false; // 跳过此帧，等待下一个 VSync
    }
}
```

#### 4.2.5 音视频同步优化

```
策略：以音频时钟为主时钟，视频追音频
```

**同步逻辑**：

```
audioClock：音频播放位置（由音频硬件驱动，最稳定）
videoClock：视频当前帧 PTS

diff = videoClock - audioClock

if (diff > 0.04s) {
    // 视频超前，延迟渲染
    delay = diff
} else if (diff < -0.04s) {
    // 视频落后，追赶
    if (diff < -0.5s) {
        // 严重落后，跳帧
        skipFramesUntil(audioClock)
    } else {
        // 轻微落后，加速渲染（缩短帧间隔）
        delay = 0
    }
} else {
    // 同步范围内，正常渲染
    delay = frameInterval
}
```

#### 4.2.6 弱网/断网场景处理

```
场景分级处理：

1. 带宽 < 播放码率（缓冲持续下降）
   → 降级到更低清晰度
   → 降低缓冲水位线，减少等待

2. 网络中断（缓冲耗尽）
   → 显示加载动画 + 已缓冲进度
   → 后台持续重试（指数退避）
   → 网络恢复后自动续播

3. 网络恢复
   → 快速填充缓冲区（2x 速度下载）
   → 缓冲恢复到低水位后开始播放
   → 逐步恢复到原清晰度
```

### 4.3 播放流畅性优化效果预估

| 优化项 | 卡顿率改善 |

## 5. 加载缓存优化

### 5.1 问题分析

- **首屏加载慢**：从点击到首帧显示耗时过长
- **重复加载**：相同视频片段反复从网络下载
- **缓存利用率低**：缓存策略不合理，命中率低
- **缓存空间浪费**：无淘汰策略，缓存无限增长
- **Seek 缓存未命中**：回看已播放内容仍需重新下载

### 5.2 优化方案

#### 5.2.1 多级缓存架构

```
┌─────────────────────────────────────────────────┐
│                   内存缓存                        │
│  L1: 解码帧缓存（已解码帧，可直接渲染）            │
│  L2: 码流缓存（已下载的原始码流数据）               │
│  容量：50~100MB                                  │
│  速度：纳秒级                                     │
├─────────────────────────────────────────────────┤
│                   磁盘缓存                        │
│  L3: 完整视频文件缓存                             │
│  L4: 分片缓存（HLS/DASH 分片）                    │
│  容量：500MB~2GB                                 │
│  速度：毫秒级                                     │
├─────────────────────────────────────────────────┤
│                   网络源                          │
│  CDN → 源站                                      │
│  速度：取决于网络                                 │
└─────────────────────────────────────────────────┘
```

#### 5.2.2 首屏加载优化

```
目标：点击到首帧显示 ≤ 1.5s

优化链路：
  T0: 用户点击
  T0+100ms: 接口请求发出（预请求可提前到 T0-500ms）
  T0+400ms: 获取播放地址，开始下载
  T0+600ms: 下载首个 GOP（关键帧组，约 0.5~2s 视频数据）
  T0+700ms: 解码器初始化完成，开始解码
  T0+900ms: 首帧解码完成，渲染显示
  T0+1000ms: 音频输出开始
  T0+1500ms: 缓冲区达到低水位，稳定播放
```

**关键优化点**：

| 阶段 | 优化措施 | 节省时间 |
|------|---------|---------|
| 接口请求 | 预请求 + DNS 预解析 | 300~500ms |
| 数据下载 | GOP 对齐请求 + CDN 就近 | 200~400ms |
| 解码初始化 | 解码器预创建 | 100~200ms |
| 首帧渲染 | 独立渲染线程 | 50~100ms |

**GOP 对齐请求**：

```
传统方式：从视频文件头开始下载，需要下载到第一个关键帧才能解码
  → 浪费时间下载非关键帧数据

优化方式：请求 CDN 从最近关键帧位置开始传输
  → HTTP Range 请求：Range: bytes={keyframe_offset}-
  → 首帧时间减少 200~500ms
```

#### 5.2.3 预加载策略

```
策略：基于用户行为的智能预加载
```

| 场景 | 预加载行为 | 预加载量 |
|------|-----------|---------|
| 视频列表滑动 | 预加载可见项的前 3s 视频 | 每个视频约 200KB |
| 连播场景 | 当前视频播放 80% 时预加载下一个 | 完整视频前 10s |
| 短视频上下滑 | 预加载上下各 1 个视频 | 完整视频 |
| 搜索结果页 | 预加载前 3 个搜索结果视频 | 前 5s |
| WiFi 环境 | 激进预加载，缓存完整视频 | 完整视频 |
| 移动网络 | 保守预加载，仅缓存首段 | 前 3~5s |

**预加载优先级**：

```
优先级排序（从高到低）：
1. 当前正在播放的视频（继续缓冲）
2. 下一集/下一个推荐视频
3. 列表中紧邻的上下视频
4. 同一播放列表中的其他视频
5. 用户历史记录中可能重看的视频

并发控制：
  - 最多同时预加载 3 个视频
  - 预加载带宽不超过总带宽的 30%
  - 不影响当前播放的下载速度
```

#### 5.2.4 磁盘缓存策略

**缓存写入策略**：

```
写入时机：
  1. 播放过程中边播边缓存（透明缓存）
  2. 预加载内容写入缓存
  3. WiFi 环境下后台缓存完整视频

写入规则：
  - 仅缓存完整分片（HLS 的 .ts 文件 / DASH 的 Segment）
  - 写入时校验完整性，不完整分片不缓存
  - 同一视频多清晰度分别缓存，低清晰度优先保留
```

**缓存淘汰策略（LRU + 优先级）**：

```
淘汰顺序（从先到后）：
1. 过期缓存（超过 TTL，默认 7 天）
2. 低优先级预加载缓存
3. 最久未访问的缓存（LRU）
4. 大文件缓存（单个视频 > 500MB）

保留规则：
  - 用户收藏/下载的视频不淘汰
  - 最近 24 小时播放过的视频不淘汰
  - 热门视频（全局排行 Top 100）保留时间延长到 30 天

空间管理：
  - 最大缓存空间：min(可用磁盘空间 * 10%, 2GB)
  - 低存储警告时（< 500MB 可用）：触发紧急淘汰
  - 缓存空间使用率 > 80%：触发常规淘汰
```

**缓存索引设计**：

```sql
CREATE TABLE video_cache (
    cache_id       TEXT PRIMARY KEY,     -- 缓存唯一标识
    video_id       TEXT NOT NULL,        -- 视频ID
    quality        TEXT NOT NULL,        -- 清晰度 (360p/480p/720p/1080p)
    file_path      TEXT NOT NULL,        -- 本地文件路径
    file_size      INTEGER NOT NULL,     -- 文件大小(bytes)
    segments       TEXT,                 -- 已缓存的分片列表(JSON)
    hit_count      INTEGER DEFAULT 0,    -- 命中次数
    last_access    INTEGER NOT NULL,     -- 最后访问时间戳
    created_at     INTEGER NOT NULL,     -- 创建时间戳
    ttl            INTEGER DEFAULT 604800, -- 生存时间(秒)，默认7天
    priority       INTEGER DEFAULT 0,    -- 优先级(0=普通,1=高,2=永久)
    is_complete    INTEGER DEFAULT 0     -- 是否完整缓存
);

-- 常用查询索引
CREATE INDEX idx_video_id ON video_cache(video_id, quality);
CREATE INDEX idx_last_access ON video_cache(last_access);
CREATE INDEX idx_priority ON video_cache(priority);
```

#### 5.2.5 缓存命中优化

```
策略：精准匹配 + 部分命中 + 回源续传
```

| 场景 | 处理方式 |
|------|---------|
| 完整命中（同视频同清晰度） | 直接从本地读取，零网络请求 |
| 部分命中（同视频，已缓存前半段） | 读取已缓存部分，仅下载缺失的分片 |
| 清晰度降级命中（请求 1080p，缓存有 720p） | 先用 720p 播放，后台下载 1080p |
| Seek 命中（回看已播放部分） | 从内存/磁盘缓存直接读取 |

**部分命中续传实现**：

```
1. 解析请求的 Segment 列表
2. 查询缓存，标记已缓存和未缓存的 Segment
3. 已缓存的 Segment 从本地读取
4. 未缓存的 Segment 从网络下载
5. 下载完成后写入缓存

  请求: [S1, S2, S3, S4, S5, S6, S7, S8]
  缓存: [S1, S2, S3,          S6, S7    ]
  结果: [本地,本地,本地,网络,网络,本地,本地,网络]
```

### 5.3 缓存优化效果预估

| 优化项 | 效果 |
|--------|------|
| 多级缓存架构 | 缓存命中率从 40%~60% 提升到 85%+ |
| 首屏加载优化 | 首屏时间从 3~5s 降低到 1.5s 以内 |
| 预加载策略 | 连播场景零等待，列表滑动零延迟 |
| 磁盘缓存策略 | 重复播放零网络请求 |
| 缓存命中优化 | 部分命中场景节省 50%~80% 带宽 |

## 6. 其他视频功能优化

### 6.1 倍速播放优化

| 问题 | 优化方案 |
|------|---------|
| 2x 以上倍速音频变调 | 使用 WSOLA/Phase Vocoder 算法保调变速 |
| 高倍速解码压力大 | 跳帧解码（只解码关键帧 + 参考帧），降低解码负载 |
| 倍速切换卡顿 | 预解码目标倍速帧，无缝切换 |

**倍速解码策略**：

```
1.0x ~ 1.5x：正常解码所有帧
1.5x ~ 2.0x：正常解码，音频 WSOLA 变速
2.0x ~ 3.0x：跳过 B 帧解码，音频 WSOLA 变速
3.0x 以上：仅解码关键帧 + P 帧，音频跳过
```

### 6.2 弹幕渲染优化

```
问题：大量弹幕时 UI 线程卡顿

优化方案：
1. 独立弹幕渲染层：使用 SurfaceView 独立图层渲染弹幕
2. 弹幕对象池：复用弹幕 View 对象，避免频繁创建/销毁
3. 弹幕合并：同一时间段相似弹幕合并显示
4. 弹幕限流：屏幕同时显示弹幕上限（默认 50 条）
5. 离屏裁剪：已移出屏幕的弹幕立即回收
```

### 6.3 画中画（PiP）优化

```
优化要点：
1. 进入 PiP 时降低分辨率到 480p，节省解码资源
2. 暂停 UI 渲染线程，仅保留视频渲染
3. 音频继续输出，不中断
4. 退出 PiP 时恢复原分辨率，无缝衔接
```

### 6.4 后台播放优化

```
优化要点：
1. 切后台时仅保留音频解码，释放视频解码资源
2. 通知栏媒体控制，支持播放/暂停/Seek
3. 音频焦点管理：来电/闹钟自动暂停
4. 内存占用控制：后台播放内存 < 50MB
```

### 6.5 字幕优化

| 问题 | 优化方案 |
|------|---------|
| 字幕加载慢 | 预加载字幕文件，与视频信息并行请求 |
| 字幕渲染卡顿 | 使用 SpannableString 替代 WebView 渲染 |
| 字幕同步偏差 | 基于音频 PTS 精确对齐，误差 < 100ms |
| 多字幕切换 | 预解析所有字幕轨道，切换时零延迟 |

### 6.6 功耗优化

```
策略：按场景动态调整性能参数

场景分级：
1. 全屏播放（最高性能）
   - 硬解码 + 高分辨率
   - 帧率 60fps
   - 缓冲积极

2. 窗口播放（平衡模式）
   - 硬解码 + 中等分辨率
   - 帧率 30fps
   - 缓冲适中

3. 后台播放（省电模式）
   - 仅音频解码
   - 释放视频资源
   - 网络按需下载

4. 低电量模式（< 20%）
   - 降级到 480p
   - 帧率 24fps
   - 减少预加载
   - 关闭弹幕
```

### 7.2 埋点方案

```
关键埋点事件：

1. play_start        - 用户点击播放
2. first_frame       - 首帧渲染完成
3. buffering_start   - 缓冲开始（卡顿开始）
4. buffering_end     - 缓冲结束（卡顿恢复）

6. seek_start        - 用户发起 Seek
7. seek_end          - Seek 完成，画面更新
8. play_error        - 播放错误
9. play_complete     - 播放完成
10. cache_hit        - 缓存命中
11. cache_miss       - 缓存未命中

每个事件携带的公共字段：
  - video_id, quality, player_type
  - network_type, bandwidth
  - device_model, os_version
  - decode_type (hard/soft)
  - timestamp, session_id
```
