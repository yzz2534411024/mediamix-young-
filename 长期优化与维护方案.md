### 1.3 最近一轮代码审查（2026-06-15）

本轮审查覆盖 35 个文件、约 8700 行新增代码，发现并修复了以下问题：

**严重问题（2个）：**
- 重复声明 `videoId` 变量导致编译失败 → 删除重复声明
- 双 `AppDatabase` 实例导致 SQLite 锁冲突 → 改为工厂单例模式

**高危问题（5个）：**
- `_getAvailableDiskSpace()` 为桩代码，磁盘满时不会触发淘汰 → 改用 `df` 命令获取真实可用空间
- `_getDeviceId()` 每次调用生成不同 ID，云端无法聚合设备数据 → 首次生成后持久化到 SharedPreferences
- 字幕下载丢失 15 秒连接超时保护 → 添加 `connectTimeout` 和 `receiveTimeout`
- WiFi 检测在带宽未初始化时误判为非 WiFi 阻断上报 → 添加缓存降级逻辑
- `privacyPreferencesProvider` 取快照后不更新，UI 展示过期数据 → 改为 `StreamProvider` 响应式

**中等及代码质量问题（7个）：**
- `_slowStartCount` Map 无界增长 → 添加 LRU 淘汰（上限 200 条目）
- `BandwidthEstimator` 类在 `network_engine.dart` 和 `player_metrics_service.dart` 中重复定义 → 合并增强版本
- `_Semaphore` 类在两处重复实现 → 改为公开类 `Semaphore`，统一引用
- Hash 算法在 `cache_service.dart` 和 `video_cache_service.dart` 中重复 → 提取到 `lib/core/utils/hash_utils.dart`
- L1/L2 内存淘汰逻辑 copy-paste → 提取通用泛型方法 `_evictOldestFromMap<T>()`
- 磁盘使用量每次遍历全量 `_diskIndex`（O(n²)） → 添加 `_cachedDiskTotalSize` 增量更新
- 内置站点 URL 在三处硬编码 → 统一使用 `CmsApiSite.defaultSites`

全部修复已合并到 main 分支，NDK 升级至 27.0.12077973，APK 构建通过（102.5MB）。

### 3.3 缓存服务深度集成

**目标**：预加载真正缓存到播放器，缓存命中时零网络请求

#### 3.3.1 播放器与缓存集成

```
当前问题：
  PreloadService 下载数据后调用 putSegment() 保存
  但 PlayerPage 打开视频时直接使用 URL，未检查本地缓存

解决方案：
  1. PlayerPage 打开视频前先查询 VideoCacheService
  2. 如果缓存命中（完整视频），使用本地文件路径替代网络URL
  3. 如果部分命中，优先使用本地分片，缺失部分从网络补充
  4. 播放过程中边播边缓存（透明代理模式）
```

```dart
// 播放器缓存集成伪代码
Future<String> _resolveVideoUrl(String url, String videoId) async {
  // 1. 检查完整缓存
  final cachedPath = await VideoCacheService.instance.getCachePath(videoId);
  if (cachedPath != null) {
    PlayerMetricsService.instance.recordEvent(MetricsEvent.cacheHit);
    return cachedPath; // 使用本地缓存
  }

  PlayerMetricsService.instance.recordEvent(MetricsEvent.cacheMiss);

  // 2. 检查CDN调度
  final cdnUrls = await _fetchCdnUrls(videoId);
  if (cdnUrls.length > 1) {
    return await VideoApiService.instance.selectBestCdn(cdnUrls);
  }

  return url;
}
```

#### 3.3.2 边播边缓存代理

```
架构：
  Player ←→ LocalProxyServer ←→ CDN

  LocalProxyServer:
    - 拦截播放器的所有数据请求
    - 已缓存分片直接返回本地数据
    - 未缓存分片从CDN下载，同时写入缓存
    - 对播放器透明，无需修改播放逻辑
```

### 4.1 多码率自适应流（HLS/DASH）

**目标**：实现真正的 ABR 码率切换

#### 4.1.1 服务端改造

```
需要服务端配合：
  1. 视频转码为多码率版本：
     - 4K (15Mbps) / 1080p (8Mbps) / 720p (4Mbps) / 480p (2Mbps) / 360p (1Mbps)
  2. 生成 HLS Master Playlist 或 DASH MPD
  3. 每个码率版本的关键帧对齐（确保无缝切换）
  4. 支持 Range 请求和 GOP 对齐请求
```

#### 4.1.2 客户端 ABR 升级

```dart
/// 增强版 ABR 控制器 — 支持真正的多码率切换
class EnhancedABRController {
  /// HLS/DASH 清单中的可用码率列表
  List<StreamRepresentation> _representations = [];

  /// 当前选中的码率索引
  int _currentIndex = 0;

  /// 码率切换决策
  StreamRepresentation? evaluate({
    required double bandwidthKbps,
    required Duration bufferDuration,
    required Duration latency,
  }) {
    // 1. 带宽过滤：排除带宽不足以支撑的码率
    final affordable = _representations
        .where((r) => r.bandwidthKbps < bandwidthKbps * 0.8)
        .toList();

    if (affordable.isEmpty) return _representations.last; // 最低码率

    // 2. 缓冲感知：缓冲不足时选择低于带宽上限的码率
    final targetBandwidth = bufferDuration < const Duration(seconds: 5)
        ? bandwidthKbps * 0.5  // 紧急降级
        : bandwidthKbps * 0.8; // 正常选择

    // 3. 选择最接近目标带宽的码率
    final selected = affordable.reduce((a, b) =>
        (a.bandwidthKbps - targetBandwidth).abs() <
        (b.bandwidthKbps - targetBandwidth).abs()
            ? a : b);

    return selected;
  }
}
```

### 4.2 服务端协同优化

#### 4.2.1 GOP 对齐请求

```
当前问题：
  播放器请求视频时从文件头开始下载，需要等到第一个关键帧才能解码
  → 浪费 200~500ms 下载非关键帧数据

服务端改造：
  1. 生成关键帧索引文件（offset → timestamp 映射）
  2. 播放器请求时携带 Range: bytes={keyframe_offset}-
  3. 服务端从最近关键帧位置开始传输

客户端改造：
  1. 获取视频信息时同时下载关键帧索引（< 10KB）
  2. 首次播放从第一个关键帧开始请求
  3. Seek 时定位到最近关键帧
```

```
               │

          │ Tracker │  协调节点
          │ Server  │  分配对等连接
          └────┬────┘
               │
          ┌────▼────┐
          │  CDN    │  回源兜底
          └─────────┘

实现要点：
  - 基于 WebRTC DataChannel 的 P2P 传输
  - 视频分片作为 P2P 交换单元
  - Tracker 服务协调对等节点
  - CDN 作为回源兜底
  - 仅在 WiFi 下启用 P2P 上传
  - 隐私保护：不暴露用户 IP
```

### 5.2 端侧 AI 优化

#### 5.2.1 智能预加载

```
基于用户行为的 AI 预加载模型：

特征：
  - 历史观看记录（时间、时长、完成率）
  - 当前播放上下文（类型、集数、时间）
  - 用户操作模式（连续观看/偶尔观看）
  - 时间特征（工作日/周末、白天/晚上）

模型：
  - 轻量级决策树（< 100KB）
  - 预测下一个可能观看的视频
  - 预测预加载的最佳时机和数量

训练：
  - 端侧联邦学习，不上传原始数据
  - 每日增量训练
  - A/B 实验验证效果
```

#### 5.2.2 画质增强

```
端侧 AI 画质增强管线：

  低码率输入 → 降噪 → 超分辨率 → 色彩增强 → 高画质输出

  各模块：
  1. 降噪：去除压缩伪影（方块效应、振铃效应）
  2. 超分辨率：720p → 1080p / 1080p → 4K
  3. 色彩增强：SDR → HDR 模拟、对比度自适应

  性能要求：
  - NPU 推理 ≤ 15ms/帧（保证 60fps）
  - 内存占用 ≤ 100MB
  - 功耗增加 ≤ 15%
```

### 5.3 跨平台统一渲染

```
目标：统一 Android/iOS/Desktop/Web 的渲染管线

方案：自研渲染层（基于 Flutter CustomPainter + Impeller）

  ┌─────────────────────────────────┐
  │        统一渲染抽象层            │

  │  弹幕渲染    │ 特效滤镜         │

  │  Impeller (GPU) / Canvas (CPU)  │

收益：
  - 消除平台差异导致的渲染问题
  - 统一 HDR 渲染管线
  - 支持自定义视频滤镜
  - 更精细的帧节奏控制
```

### 6.4 版本发布节奏

```

  Hotfix（紧急修复）：

  灰度发布流程：
    5% → 观察2天 → 10% → 观察2天 → 30% → 观察3天 → 50% → 100%
```

### 9.2 模块化拆分计划

```
Step 1: 提取 PlayerCore
  - 将播放器核心逻辑从 PlayerPage 中提取
  - PlayerCore 作为独立类，管理 Player 实例和所有引擎
  - PlayerPage 仅负责 UI 渲染

Step 2: 定义引擎接口
  - DecodeEngine (硬解/软解/超分)
  - CacheEngine (四级缓存/边播边缓存)
  - ABREngine (码率决策/流切换)
  - NetworkEngine (DNS/CDN/重试)
  - MetricsEngine (埋点/告警/上报)

Step 3: 引擎可替换
  - 每个引擎通过接口定义，可独立替换实现
  - 支持运行时切换引擎实现（A/B 测试）
  - 引擎间通过事件总线通信
```

## 10. 风险与应对

### 10.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| media_kit 重大版本升级 API 不兼容 | 中 | 高 | 锁定版本，充分测试后升级 |
| 硬解码兼容性问题（特定机型） | 高 | 中 | 完善解码能力探测，自动降级 |
| 内存泄漏（长时播放） | 中 | 高 | 定期内存巡检，DevTools 分析 |
| iOS 审核被拒（视频源合规） | 低 | 高 | 内容审核机制，合规声明 |
| P2P 传输安全风险 | 低 | 高 | 加密传输，仅 WiFi 上传 |

### 10.2 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| CDN 成本增长 | 高 | 中 | P2P 分流，缓存命中率提升 |
| 用户增长导致服务端压力 | 中 | 高 | 弹性扩容，客户端缓存分担 |
| 竞品功能追赶 | 中 | 中 | 持续优化用户体验，建立技术壁垒 |
| 隐私合规要求变更 | 低 | 高 | 数据最小化采集，联邦学习 |

### 10.3 应急预案

```
P0 级故障（播放完全不可用）：
  1. 5分钟内确认故障范围
  2. 10分钟内启动回滚或热修复
  3. 30分钟内恢复服务
  4. 24小时内完成故障复盘

P1 级故障（部分功能异常）：
  1. 30分钟内确认影响范围
  2. 2小时内发布修复版本
  3. 48小时内完成故障复盘

P2 级故障（体验降级）：
  1. 4小时内确认问题
  2. 下个版本修复
  3. 周度复盘
```
