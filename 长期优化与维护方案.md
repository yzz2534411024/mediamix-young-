### 1.4 最近一轮代码审查（2026-06-15）

本轮审查覆盖 35 个文件、约 8700 行新增代码，发现并修复了以下问题：

**严重问题（2个）：**
- 重复声明 `videoId` 变量导致编译失败 → 删除重复声明
- 双 `AppDatabase` 实例导致 SQLite 锁冲突 → 改为工厂单例模式

**高危问题（5个）：**
- `_getAvailableDiskSpace()` 为桩代码，磁盘满时不会触发淘汰 → 改用 `df` 命令获取真实可用空间
- `_getDeviceId()` 每次调用生成不同 ID，云端无法聚合设备数据 → 首次生成后持久化到 SharedPreferences
- 字幕下载丢失 15 秒连接超时保护 → 添加 `connectTimeout` 和 `receiveTimeout`
- WiFi 检测在带宽未初始化时误判为非 WiFi 阻断上报 → 添加缓存降级逻辑
- `privacyPreferencesProvider` 取快照后不更新，UI 展示过期数据 → 改为 `StreamProvider` 响应式

**中等及代码质量问题（7个）：**
- `_slowStartCount` Map 无界增长 → 添加 LRU 淘汰（上限 200 条目）
- `BandwidthEstimator` 类在 `network_engine.dart` 和 `player_metrics_service.dart` 中重复定义 → 合并增强版本
- `_Semaphore` 类在两处重复实现 → 改为公开类 `Semaphore`，统一引用
- Hash 算法在 `cache_service.dart` 和 `video_cache_service.dart` 中重复 → 提取到 `lib/core/utils/hash_utils.dart`
- L1/L2 内存淘汰逻辑 copy-paste → 提取通用泛型方法 `_evictOldestFromMap<T>()`
- 磁盘使用量每次遍历全量 `_diskIndex`（O(n²)） → 添加 `_cachedDiskTotalSize` 增量更新
- 内置站点 URL 在三处硬编码 → 统一使用 `CmsApiSite.defaultSites`

全部修复已合并到 main 分支，NDK 升级至 27.0.12077973，APK 构建通过（102.5MB）。

## 4. 短期优化（1~3个月）

> 调整说明：短期优先级从"继续推进新功能"转向"补齐关键短板+测试体系"，确保已实现功能的稳定性和可用性。

### 4.1 缓存服务与播放器深度集成（P0）

**目标**：预加载真正生效，缓存命中时零网络请求

**当前问题**：
```
PreloadService 下载数据后调用 putSegment() 保存
但 PlayerPage 打开视频时直接使用 URL，未检查本地缓存
→ 预加载功能形同虚设
```

#### 4.1.1 播放器缓存读取

```dart
// 播放器缓存集成实现
Future<String> _resolveVideoUrl(String url, String videoId) async {
  // 1. 检查完整缓存
  final cachedPath = await VideoCacheService.instance.getCachePath(videoId);
  if (cachedPath != null) {
    PlayerMetricsService.instance.recordEvent(MetricsEvent.cacheHit);
    return cachedPath; // 使用本地缓存
  }

  PlayerMetricsService.instance.recordEvent(MetricsEvent.cacheMiss);

  // 2. 检查CDN调度
  final cdnUrls = await _fetchCdnUrls(videoId);
  if (cdnUrls.length > 1) {
    return await VideoApiService.instance.selectBestCdn(cdnUrls);
  }

  return url;
}
```

#### 4.1.2 边播边缓存代理

```
架构：
  Player ←→ LocalProxyServer ←→ CDN

  LocalProxyServer:
    - 拦截播放器的所有数据请求
    - 已缓存分片直接返回本地数据
    - 未缓存分片从CDN下载，同时写入缓存
    - 对播放器透明，无需修改播放逻辑
```

#### 4.1.3 降级策略完整链路

```
重试全部失败 → 尝试低清晰度播放地址
低清晰度也失败 → 检查本地缓存（任何清晰度）
缓存也未命中 → 展示错误页，提供重试按钮
```

#### 5.1.2 客户端 ABR 升级

```dart
/// 增强版 ABR 控制器 — 支持真正的多码率切换
class EnhancedABRController {
  /// HLS/DASH 清单中的可用码率列表
  List<StreamRepresentation> _representations = [];

  /// 当前选中的码率索引
  int _currentIndex = 0;

  /// 码率切换决策
  StreamRepresentation? evaluate({
    required double bandwidthKbps,
    required Duration bufferDuration,
    required Duration latency,
  }) {
    // 1. 带宽过滤：排除带宽不足以支撑的码率
    final affordable = _representations
        .where((r) => r.bandwidthKbps < bandwidthKbps * 0.8)
        .toList();

    if (affordable.isEmpty) return _representations.last; // 最低码率

    // 2. 缓冲感知：缓冲不足时选择低于带宽上限的码率
    final targetBandwidth = bufferDuration < const Duration(seconds: 5)
        ? bandwidthKbps * 0.5  // 紧急降级
        : bandwidthKbps * 0.8; // 正常选择

    // 3. 选择最接近目标带宽的码率
    final selected = affordable.reduce((a, b) =>
        (a.bandwidthKbps - targetBandwidth).abs() <
        (b.bandwidthKbps - targetBandwidth).abs()
            ? a : b);

    return selected;
  }
}
```

### 10.2 模块化拆分计划

```
Step 1: 提取 PlayerCore（短期，与4.3节同步）
  - 将播放器核心逻辑从 PlayerPage 中提取
  - PlayerCore 作为独立类，管理 Player 实例和所有引擎
  - PlayerPage 仅负责 UI 渲染

Step 2: 定义引擎接口（中期）
  - DecodeEngine (硬解/软解/超分)
  - CacheEngine (四级缓存/边播边缓存)
  - ABREngine (码率决策/流切换)
  - NetworkEngine (DNS/CDN/重试)
  - MetricsEngine (埋点/告警/上报)

Step 3: 引擎可替换（长期）
  - 每个引擎通过接口定义，可独立替换实现
  - 支持运行时切换引擎实现（A/B 测试）
  - 引擎间通过事件总线通信
```

### 11.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| media_kit 重大版本升级 API 不兼容 | 中 | 高 | 锁定版本，充分测试后升级 |

| 缓存集成后播放器行为异常 | 中 | 高 | 充分测试，灰度发布，保留回退路径 |

| 风险 | 概率 | 影响 | 应对措施 |

### 11.3 应急预案

```
P0 级故障（播放完全不可用）：

  3. 30分钟内恢复服务
  4. 24小时内完成故障复盘

P1 级故障（部分功能异常）：
  1. 30分钟内确认影响范围
  2. 2小时内发布修复版本
  3. 48小时内完成故障复盘

P2 级故障（体验降级）：
  1. 4小时内确认问题

  3. 周度复盘
```
